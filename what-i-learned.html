What I Learned

Reflections on transitioning from frontend mechanics to multi-tenant backend architecture while building the Issue Tracker API.

01

The Defensive Hashing Mindset

On the frontend, security is about hiding buttons. On the backend, the network is hostile, and plain-text passwords are a critical vulnerability. Hashing is strict one-way encryption.

Using Argon2 ensures protection because it is a memory-hard algorithm. Attackers cannot easily brute-force the hashes even if they steal the database because the function is deliberately slow and resource-intensive.

  • Never store passwords in plain text.
  • Salting and stretching are handled automatically inside Argon2 to prevent rainbow table attacks.
  • Always return generic login messages ("Invalid email or password") to block email enumeration attacks.
02

Decoupling Database from Domain (Anti-Corruption)

Tying the rest of the application directly to the ORM (Prisma) generated types creates tight coupling. If the database schema changes, the entire application breaks.

Creating Domain Types (like custom User interfaces) and a serialization utility (`sanitizeUser`) forms an anti-corruption layer. Only the database layer knows about Prisma; the rest of the app operates on pure domain models.

  • Exposing database schemas directly to controllers leaks internal details.
  • Sanitizing records (like removing `passwordHash`) ensures sensitive fields are never leaked to HTTP clients.
  • Makes swapping database layers or schemas in the future far easier.
03

Separation of Layered Responsibilities

Structuring our monolith into specific modules and layers keeps handlers thin and boundaries clean. Every layer has exactly one job:

  • Router: Maps paths and chains edge middlewares (Validation, Authentication, Authorization).
  • Middleware: Intercepts requests for cross-cutting tasks (validating schemas, fetching session state, asserting roles).
  • Controller: Translates HTTP. It parses route parameters, executes requests, sets cookies, and formats JSON.
  • Service: Houses core business logic and performs relational checks.
  • Repository: Executes raw data access (Prisma / SQL), hiding query implementation details.
04

Authentication vs. Authorization Scoping

Knowing who the user is (Authentication) is only half the battle. Checking what workspace resources they can access (Authorization) is crucial for multi-tenant isolation.

Using Express nested routers with `mergeParams: true` allowed us to build hierarchical access control. We check workspace membership once at the parent mount point, protecting all nested sub-routes (projects, issues, comments).

  • `requireAuth` verifies the session exists.
  • `requireWorkspaceMember` extracts `:workspaceId`, verifies membership, and populates `req.membership`.
  • `requireWorkspaceRole` checks permissions (e.g. blocking viewers from writing).
  • Resource scope checks verify that projects and issues belong to the target workspace to prevent cross-tenant leaks.
05

Database Invariants & API Usability

Good API list behavior and transactional integrity are crucial for a professional-grade backend.

  • Database Transactions: Wrapped workspace creation and admin membership creation in an atomic transaction (`prisma.$transaction`) to prevent orphaned workspaces.
  • Stable Ordering: Appended a unique ID fallback (`{ id: 'asc' }`) to all query sorting arrays. This prevents rows from shifting randomly between pages as a user scrolls.
  • Offset Pagination: Implemented parallel queries (`Promise.all`) running a count query and a limit/skip query to return predictable metadata (`{ data, meta: { page, limit, total } }`).
  • Soft Deletion: Configured project and issue deletions as soft updates (setting status to `'archived'`) to preserve database history.
